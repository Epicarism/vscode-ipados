### 3. Why They Might Be Nil When a Folder IS Open

There is a **two-layer optional chain** (`fileNavigator?.rootURL`) where EITHER layer can be nil:

**Layer 1 — `fileNavigator` itself is nil:**
- `AIToolExecutor.fileNavigator` is declared as **`weak var fileNavigator: FileSystemNavigator?`** (AIAgentTools.swift line 239). A weak reference can become nil if the referenced object is deallocated.
- `AIAssistantView.fileNavigator` is `var fileNavigator: FileSystemNavigator?` (AIAssistantView.swift line 8) — just a plain optional.
- **Critical:** `toolExecutor` is a **computed property** (AIAssistantView.swift lines 16-18) that creates a **new `AIToolExecutor` instance every time it's accessed**:
  ```swift
  private var toolExecutor: AIToolExecutor {
      AIToolExecutor(editorCore: editorCore, fileNavigator: fileNavigator)
  }
  ```
  The new executor holds only a `weak` reference back to `fileNavigator`. If the executor is captured in a closure or Task that outlives the view's evaluation, and no strong reference keeps `fileNavigator` alive at that moment, the weak reference becomes nil.

**Layer 2 — `rootURL` itself is nil:**
- `rootURL` starts as `nil` and is only set when `loadFileTree(at:)` is called (line 17).
- If the user opened a folder but `loadFileTree` hasn't been called yet (e.g., trust dialog is still pending at ContentView.swift line 36), `rootURL` remains nil.
- `refreshFileTree()` (line 69) reads from `fileTree.url` rather than `rootURL`, meaning a code path could exist where `fileTree` is set but `rootURL` isn't (though currently `loadFileTree` sets both).

**Most Likely Real-World Scenario:** The `weak var` on `AIToolExecutor` is the primary risk. Since the executor is recreated as a computed property and used inside `async` Tasks, there's a window where the weak reference could be nil even though `ContentView`'s `@StateObject` still owns the navigator. The fix would be to either make the reference `strong` (non-weak) in `AIToolExecutor`, or capture `fileNavigator` strongly before async work.
