21|821276|## JavaScriptCore (JSC) on iOS: capabilities & limitations
22|adc83b|
23|e785fc|### What JSC is good for
24|adc83b|
25|77c3fc|JavaScriptCore (`JavaScriptCore.framework`) is a good fit for **embedded, sandboxed scripting**:
26|adc83b|
27|cea617|- User scripting / automation (macros, editor actions)
28|9ef391|- Lightweight formatters, parsers, validators
29|d57497|- Rule engines / expression evaluation
30|08a395|- “Extension-like” logic **if** you expose a minimal, audited native API surface
31|adc83b|
32|9e5742|### Key limitations (important for “VS Code on iPad”)
33|adc83b|
34|eb1b5a|1. **Not a web runtime (no browser APIs)**
35|f27458|   - No DOM.
36|dd44bf|   - No `fetch`, `XMLHttpRequest`, `WebSocket`, etc.
37|d53a87|   - Networking, filesystem, and device access must be implemented in native code and bridged into JS.
38|adc83b|
39|e10919|2. **JIT is generally not available to normal App Store apps**
40|00d689|   - iOS code-signing + W^X-style restrictions mean third-party apps typically cannot use JIT without special entitlements.
41|447ab2|   - Consequence: CPU-heavy JS workloads can be significantly slower on device than on macOS.
42|adc83b|
43|35e6c6|3. **No Node-style module ecosystem**
44|8a9ffd|   - There is no `require('fs')`/`npm` environment.
45|df465c|   - If you want modules, you implement a loader yourself and strictly control what “imports” can do.
46|adc83b|
47|33ce31|4. **Resource exhaustion controls are on you**
48|8da655|   - Embedded JSC does not provide a universal “watchdog” timeout.
49|56f661|   - You must implement timeouts/cancellation and memory-pressure strategies at the host level.
50|adc83b|
51|0a074b|### Practical recommendation
52|adc83b|
53|6c0c94|Use JSC for:
54|c115d8|- small, user-visible scripting surfaces
55|9cc61c|- deterministic automation
56|4f3ec6|- running code that can be paused/limited and that doesn’t need full OS/toolchain access
57|adc83b|

119|bd5572|## Why Python can’t run natively (no interpreter allowed)
120|adc83b|
121|663838|For this project, treat **general-purpose Python execution on-device as disallowed**.
122|adc83b|
123|cf271d|There are two reasons—**App Store policy** (the gating issue for distribution) and **platform constraints** (which make “desktop Python” assumptions invalid anyway).
124|adc83b|
125|ef5a72|### 1) App Store policy: downloading/executing code that changes app behavior
126|adc83b|
127|b7a139|Apple App Review Guideline **2.5.2** states:
128|adc83b|
129|699337|> “Apps should be self-contained in their bundles, and may not read or write data outside the designated container area, nor may they download, install, or execute code which introduces or changes features or functionality of the app, including other apps.”
130|adc83b|
131|fd238b|It also includes a narrow exception for some educational use cases:
132|adc83b|
133|0d598f|> “Educational apps designed to teach, develop, or allow students to test executable code may, in limited circumstances, download code… Such apps must make the source code provided by the app completely viewable and editable by the user.”
134|adc83b|
135|3833ae|Source:
136|e30b03|- https://developer.apple.com/app-store/review/guidelines/#software-requirements
137|adc83b|
138|249d24|**Implication for VSCodeiPadOS:**
139|bdf9ca|- A bundled Python interpreter (plus package installation, plus arbitrary user code) can be interpreted as a mechanism to add/alter functionality after review.
140|15ab57|- Even if you could argue an “educational” exception, that typically requires strict UX constraints (source must be viewable/editable) and still doesn’t solve the rest of the platform limitations below.
141|adc83b|
142|fdedac|### 2) Platform constraints (even if policy were satisfied)
143|adc83b|
144|e1076f|Even embedding CPython in-process doesn’t create a desktop-like environment:
145|adc83b|
146|a663eb|- **No arbitrary process execution model** (you can’t just run downloaded toolchain binaries; the iOS app sandbox does not behave like macOS/Linux).
147|4726e6|- **Native extension modules are a problem** (typical `pip` wheels include native code; loading arbitrary native code post-review is a non-starter).
148|92d329|- **No JIT for third-party apps** (relevant to PyPy/JIT and some performance strategies), due to iOS code-signing and W^X-style restrictions.
149|adc83b|
150|47377f|### Practical decision
151|adc83b|
152|1ed491|- Don’t try to be a self-contained Python workstation on iPad.
153|053db2|- Provide Python (and other toolchains) via **remote execution** (SSH / remote compute).

119|bd5572|## Why Python can’t run natively (no interpreter allowed)
120|adc83b|

122|adc83b|

126|adc83b|
127|b7a139|Apple App Review Guideline **2.5.2** states:
128|adc83b|

115|bd5572|## Why Python can’t run natively (no interpreter allowed)
116|adc83b|
117|663838|For this project, treat **general-purpose Python execution on-device as disallowed**.
118|adc83b|
119|cf271d|There are two reasons—**App Store policy** (the gating issue for distribution) and **platform constraints** (which make “desktop Python” assumptions invalid anyway).
120|adc83b|
121|ef5a72|### 1) App Store policy: downloading/executing code that changes app behavior
122|adc83b|
123|b7a139|Apple App Review Guideline **2.5.2** states:
124|adc83b|
125|699337|> “Apps should be self-contained in their bundles, and may not read or write data outside the designated container area, nor may they download, install, or execute code which introduces or changes features or functionality of the app, including other apps.”
126|adc83b|
127|fd238b|It also includes a narrow exception for some educational use cases:
128|adc83b|
129|0d598f|> “Educational apps designed to teach, develop, or allow students to test executable code may, in limited circumstances, download code… Such apps must make the source code provided by the app completely viewable and editable by the user.”
130|adc83b|
131|249d24|**Implication for VSCodeiPadOS:**
132|bdf9ca|- A bundled Python interpreter (plus package installation, plus arbitrary user code) can be interpreted as a mechanism to add/alter functionality after review.
133|15ab57|- Even if you could argue an “educational” exception, that typically requires strict UX constraints (source must be viewable/editable) and still doesn’t solve the rest of the platform limitations below.
134|adc83b|
135|fdedac|### 2) Platform constraints (even if policy were satisfied)
136|adc83b|
137|e1076f|Even embedding CPython in-process doesn’t create a desktop-like environment:
138|adc83b|
139|a663eb|- **No arbitrary process execution model** (you can’t just run downloaded toolchain binaries; the iOS app sandbox does not behave like macOS/Linux).
140|4726e6|- **Native extension modules are a problem** (typical `pip` wheels include native code; loading arbitrary native code post-review is a non-starter).
141|92d329|- **No JIT for third-party apps** (relevant to PyPy/JIT and some performance strategies), due to iOS code-signing and W^X-style restrictions.
142|adc83b|
143|47377f|### Practical decision
144|adc83b|
145|1ed491|- Don’t try to be a self-contained Python workstation on iPad.
146|053db2|- Provide Python (and other toolchains) via **remote execution** (SSH / remote compute).
147|adc83b|
